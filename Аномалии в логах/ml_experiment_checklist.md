### Гипотеза

> **Isolation Forest** по engineered-признакам временного ряда **лучше справится** с поиском аномалий в веб-трафике, чем статистические методы (**Z-score / IQR**), по метрике **F1** на test-выборке.

| Элемент | Описание |
| :--- | :--- |
| **Target** | `is_anomaly` (0/1) — аномальный интервал трафика; на выходе модели также `if_score` |
| **Proxy-разметка** | Готовых меток в датасете нет; для оценки используем Z-score как псевдо-разметку (ограничение — проговорить на защите) |
| **Features (исходные)** | `Timestamp`, `TrafficCount` |
| **Features (engineered)** | `hour`, `day_of_week`, `traffic_lag_1`, `traffic_pct_change`, `rolling_mean`, `rolling_std`, `z_score` |
| **Baseline** | Z-score (отклонение от скользящего среднего > N·σ) и IQR (выход за границы квартилей в окне) |
| **ML-модель** | Isolation Forest (unsupervised) |
| **Основная метрика** | Precision, Recall, **F1** — классы несбалансированы (~2–5% аномалий), Accuracy не подходит |

### Обоснование Baseline

| Метод | Почему выбран |
| :--- | :--- |
| **Z-score** | Одна метрика, понятное правило, быстрый старт, легко объяснить на защите |
| **IQR** | Устойчивее к выбросам внутри окна, альтернативный статистический baseline |
| **Не ML** | Z-score / IQR — статистика, не модель; нужны как точка сравнения для IF |

#### Кросс-валидация по времени — что это и зачем

Сейчас в проекте **один** хронологический split 80/20 (`train_ratio` в `config.yaml`): модель обучается на первых 80% ряда и проверяется на последних 20%. Оценка качества зависит от того, **какой именно** отрезок попал в test.

**Кросс-валидация по времени** — несколько таких разрезов подряд: на каждом шаге train всегда **раньше**, validation/test — **следующий** по времени кусок. Данные **не перемешиваются** (в отличие от обычной k-fold CV).

```
Ряд:     |----|----|----|----|----|----|----|----|

Фолд 1:  [train] [val]
Фолд 2:  [train----] [val]
Фолд 3:  [train--------] [val]
Фолд 4:  [train------------] [val]
```

В sklearn это `TimeSeriesSplit`: на каждом фолде заново обучаем IF (и baseline), считаем F1/Precision/Recall, затем усредняем метрики по фолдам.

| | Один split 80/20 (сейчас) | CV по времени |
| :--- | :--- | :--- |
| Оценка | Одна цифра на последних 20% | Среднее по нескольким «будущим» окнам |
| Риск | Метрика может «повезти» с конкретным периодом | Более устойчивая оценка на новых данных |
| Утечка из будущего | Нет (split по `Timestamp`) | Нет, если не shuffle |
| Сложность | Проще, быстрее | Дольше: обучение × число фолдов |

**Почему не shuffle:** если случайно перемешать строки временного ряда, в train попадут точки **после** test — модель увидит будущее и метрики будут завышены.

**Как применить к нашему пайплайну (идея, не реализовано):**

1. Разбить отсортированный `df` на 4–5 фолдов `TimeSeriesSplit`.
2. На каждом фолде: rolling/lag/features → обучить IF → сравнить с Z-score/IQR на validation-окне.
3. Зафиксировать среднее и разброс F1 — если разброс большой, выводы по одному split 80/20 менее надёжны.

### Результаты на test (proxy-метки Z-score)

Вывод `python -m src.models.train`:

| Метод | Precision | Recall | F1 |
| :--- | :---: | :---: | :---: |
| IQR | 0.579 | 1.000 | 0.733 |
| IF (contamination) | 1.000 | 0.273 | 0.429 |
| **IF + порог (итерация 2)** | **0.889** | 0.727 | **0.800** |

### Почему выбраны эти метрики

| Метрика | Смысл для алертинга |
| :--- | :--- |
| **Precision** | Доля реальных аномалий среди сработавших алертов (меньше шума для дежурного) |
| **Recall** | Доля найденных аномалий от всех реальных (меньше пропусков) |
| **F1** | Баланс Precision и Recall при дисбалансе классов |
| ~~Accuracy~~ | Не используем: при 2% аномалий модель «всё норма» даст 98% accuracy |

### Две итерации эксперимента

| Итерация | Что меняли | Результат |
| :--- | :--- | :--- |
| **1** | Isolation Forest с `contamination` | Высокий Precision, низкий Recall — слишком мало алертов на test |
| **2** | Подбор порога `if_score` по перцентилям на train | F1 вырос с 0.43 → **0.80**, Precision 0.89 при Recall 0.73 |

### Выводы по гипотезе

**Гипотеза подтвердилась частично.**

- IF с `contamination` **проиграл** IQR по F1 — гипотеза **не подтвердилась** в итерации 1.
- IF с подобранным порогом **превзошёл** IQR по F1 (0.80 vs 0.73) и Precision — гипотеза **подтвердилась** в итерации 2.
- IQR по-прежнему лучше по **Recall** (1.0): ловит все Z-score-аномалии, но с большим числом ложных тревог.

### Почему модель ошибается / ограничения

1. **Нет истинной разметки** — оценка через proxy Z-score смещает сравнение в пользу статистических методов.
2. **Одна метрика** — реальные логи содержат IP, коды ответов, текст; датасет упрощён.
3. **Сезонность** — дневные пики могут выглядеть как аномалия (частично учтено через `hour`, `day_of_week`).
4. **Contamination** на train не всегда переносится на test — порог по score решает эту проблему.

### Что улучшили (итерация 2)

- Перешли от фиксированного `contamination` к **явному порогу** на `if_score`.
- Порог подбирается на **train** (без утечки из test).
- Recall вырос с 0.27 → 0.73 при сохранении высокого Precision.
- Код вынесен в воспроизводимый модульный пайплайн с сохранением артефактов.
